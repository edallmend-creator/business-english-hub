# Supabase Launch Checklist

## Current App Model

- The hub uses a pseudonymous progress code, not e-mail login.
- `js/sync.js` writes to Supabase through RPC functions with the public anon key.
- Stored data is learning progress only: completed exercises, SRS reviews, streaks, settings and session logs.
- The public anon key is expected in a static frontend. It is not a secret. Security must come from RLS, revoked table permissions, and controlled RPC functions.

## Important Security Note

The progress code acts like a lightweight password for the learning state. It should be reasonably hard to guess, easy to copy, and should not contain personal data.

Before launch, keep codes easy enough for users to write down. The current format is `word-12345`, which is a pragmatic tradeoff because the stored data is low-risk learning progress.

## Tables Used By The Frontend

### `user_progress`

Required columns:

- `user_code` text primary key
- `completed_exercises` jsonb default `{}`
- `dual_coding` jsonb default `{}`
- `settings` jsonb default `{}`
- `flashcard_positions` jsonb default `{}`
- `streak_current` integer default `0`
- `streak_best` integer default `0`
- `streak_last_date` date nullable
- `total_reviews` integer default `0`
- `total_correct` integer default `0`
- `updated_at` timestamptz default `now()`

### `card_reviews`

Required columns:

- `user_code` text not null
- `card_id` text not null
- `deck` text nullable
- `ease_factor` numeric default `2.5`
- `interval_days` integer default `1`
- `repetitions` integer default `0`
- `next_review` date nullable
- `last_reviewed` timestamptz nullable
- `correct_count` integer default `0`
- `wrong_count` integer default `0`
- `again_count` integer default `0`
- primary key: (`user_code`, `card_id`)

### `session_log`

Required columns:

- `id` bigint generated identity primary key
- `user_code` text nullable
- `exercise_type` text nullable
- `deck` text nullable
- `cards_reviewed` integer default `0`
- `cards_correct` integer default `0`
- `duration_sec` integer default `0`
- `created_at` timestamptz default `now()`

## Minimum SQL Setup

Use `docs/SUPABASE_CODE_SYNC_RPC.sql` for the current production model. It creates/updates the required tables, enables RLS, revokes direct table access and grants only controlled RPC functions.

Older direct table access from the browser is intentionally avoided.

Reference table setup:

```sql
create table if not exists public.user_progress (
  user_code text primary key,
  completed_exercises jsonb not null default '{}'::jsonb,
  dual_coding jsonb not null default '{}'::jsonb,
  settings jsonb not null default '{}'::jsonb,
  flashcard_positions jsonb not null default '{}'::jsonb,
  streak_current integer not null default 0,
  streak_best integer not null default 0,
  streak_last_date date,
  total_reviews integer not null default 0,
  total_correct integer not null default 0,
  updated_at timestamptz not null default now()
);

create table if not exists public.card_reviews (
  user_code text not null,
  card_id text not null,
  deck text,
  ease_factor numeric not null default 2.5,
  interval_days integer not null default 1,
  repetitions integer not null default 0,
  next_review date,
  last_reviewed timestamptz,
  correct_count integer not null default 0,
  wrong_count integer not null default 0,
  again_count integer not null default 0,
  primary key (user_code, card_id)
);

create table if not exists public.session_log (
  id bigint generated by default as identity primary key,
  user_code text,
  exercise_type text,
  deck text,
  cards_reviewed integer not null default 0,
  cards_correct integer not null default 0,
  duration_sec integer not null default 0,
  created_at timestamptz not null default now()
);

alter table public.user_progress enable row level security;
alter table public.card_reviews enable row level security;
alter table public.session_log enable row level security;
```

## Policy Decision Before Launch

With the current no-login architecture, strict per-user RLS is not possible from direct browser table access because all visitors use the same `anon` role. Therefore direct table access should remain blocked and sync should go through RPC functions.

There are two realistic options:

### Option A: Accept Pseudonymous Code-Based RPC Sync

Use memorable progress codes and store no personal data. This is acceptable for low-risk learning progress, but a guessed/shared code can access that progress.

Direct table access remains blocked. RPC functions only return/update rows for the supplied progress code.

### Option B: Stronger Security

Move sync behind Supabase Auth, RPC functions, or Edge Functions. Do not allow direct anon table reads. This is more work, but gives real ownership controls and rate limiting options.

Recommended long-term approach: anonymous Supabase Auth plus server-side restore/import flow for cross-device progress codes.

## Pre-Launch Checks

- Confirm RLS is enabled on all three tables.
- If Supabase reports `rls_disabled_in_public`, immediately run `docs/SUPABASE_CODE_SYNC_RPC.sql` in the Supabase SQL Editor.
- Confirm no service-role key appears in frontend files.
- Confirm no e-mail, name, company or other personal profile data is written to progress tables.
- Test a new browser profile: progress code appears, progress saves, reload restores.
- Test switching to an existing code restores the same progress.
- Test offline mode: app still works if Supabase is unreachable.
- Check Supabase logs for failed RLS queries before launch.
- Set a low-risk retention policy manually if desired, e.g. delete inactive session logs after a defined period.
